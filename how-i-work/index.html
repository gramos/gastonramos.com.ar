I value an engineering culture that minimizes unnecessary meetings, focuses on building and shipping, and prioritizes speed and pragmatism. I'm a fan of Extreme Programming, so these principles resonate strongly with me.

I'm especially interested in technical products built for engineers, including areas related to observability, developer tooling, infrastructure, and production systems.

A Project I'm Proud Of

The project I'm most proud of from my recent role was the Rules Engine at Clio.

It was a system to automate accounting actions on bank transactions based on different criteria like amount, transaction type, merchant, description, and other attributes.

I built the initial prototype and helped define the architecture. After that, I led the implementation with a small team of engineers.

What made it interesting was that the problem was not only technical. It was a product problem too. We had to make something flexible enough for real accounting workflows, but still simple and understandable for users.

There were also hard backend problems around background jobs, transaction states, consistency, and making sure asynchronous processes didn't overwrite each other.

I'm proud of it because it combined many things I enjoy: product thinking, backend architecture, simplicity, and leading a project from prototype to production.

How I Use AI

I use AI for programming, building prototypes, reviewing pull requests, creating project plans, and writing tickets.

My normal workflow is the following: first, I plan what I need to do and try to be as precise as possible when writing the prompt. Once I have the plan, I save it and use it as the basis for moving forward with the code.

I review the result critically, and if I find something I'm not happy with, I ask the agent to make the necessary changes. I generally try to simplify the solution until I find the simplest approach that can work correctly.

When I finish, I use a separate agent to review what I did and help me write the tests. After that, I review and run the tests, and I manually verify that the behavior is what I expected.

My General Coding Problem-Solving Process

  1. Understand the problem

    • Identify the input and expected output.
    • Restate the problem in my own words.
    • Review the provided examples.
    • Identify ambiguities and constraints.
    • Note the most important edge cases.

    What transformation must happen between the input and the output?

  2. Design a simple solution

    • Solve one example manually.
    • Choose an appropriate data structure, such as an array, hash, set, stack, or queue.
    • Start with the simplest correct solution.
    • Estimate time and space complexity.
    • Avoid optimizing before the problem is understood.

  3. Define test cases

    • A normal example.
    • Empty or minimum input.
    • Duplicates.
    • Zero or negative values when relevant.
    • A case that could break a naive solution.

  4. Implement

    • Use a small method with a descriptive name.
    • Use clear variable names.
    • Avoid premature abstractions.
    • Prefer standard language data structures and methods.
    • Do not mutate the input unless intentional.
    • Handle meaningful edge cases explicitly.
    • Keep the main logic separate from test or demonstration code.

  5. Test and debug

    • Run the examples from the problem.
    • Run additional edge cases.
    • Read the complete error before changing code.
    • Check indexes, termination conditions, and unexpected null or nil values.
    • Verify that the solution does not accidentally depend on ordering.
    • Understand the cause of a failure before patching it.

  6. Review and explain

    • Re-read the problem.
    • Confirm the return type and exact output.
    • Remove debugging and dead code.
    • Simplify unclear names or conditions.
    • Explain why the solution is correct.
    • State time and space complexity.
    • Mention limitations honestly.

First make it correct, then clear, and only then optimize it.

Explaining My Thought Process

I communicate decisions at meaningful checkpoints rather than narrating every keystroke.

  1. Confirm my interpretation of the problem.
  2. Describe the initial approach.
  3. Explain the selected data structure.
  4. Mention important edge cases.
  5. Implement without talking continuously.
  6. Explain what each test validates.
  7. Finish with correctness and complexity.

Example phrases

  • “Let me first make sure I understand the problem correctly.”
  • “My first approach would be to...”
  • “I think a hash would be useful here because...”
  • “Before coding, I want to check a couple of edge cases.”
  • “This solution requires one pass through the input.”
  • “The time complexity is O(n), and the space complexity is O(n).”
  • “Let me test it with the example from the prompt.”
  • “I see a simpler way to express this condition.”

Changing direction based on new evidence demonstrates judgment rather than failure.

AI During Problem Solving

  • Use AI to review the interpretation, identify edge cases, or challenge the proposed complexity.
  • Never accept code I cannot explain and reconstruct independently.
  • Review every generated change.
  • Execute tests and verify behavior manually.
  • Retain ownership of the design and final technical decisions.
